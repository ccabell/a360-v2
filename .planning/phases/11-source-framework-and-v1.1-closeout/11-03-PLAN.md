# Enrichment Pipeline: Repeatable Source-to-Fuel Loop

**Phase:** 11 — Source Framework & v1.1 Close-Out
**Status:** Governing document — follow this pipeline for all enrichment tasks in v1.2+
**Requirements:** SRCE-03
```

**Section 1: Pipeline Overview**

The enrichment pipeline is a repeatable loop, not a one-time build. Every new source (IFU, PubMed paper, manufacturer video, YouTube channel, industry article) follows the same 5-step path from raw material to searchable evidence to fuel doc content.

```
[New Source] → Classify → Chunk → Vector DB → Mark Fuel Stale → Regenerate
     ↑                                                              |
     └──────────────── Add more sources ────────────────────────────┘
```

**Section 2: Step 1 — Add Source**

What triggers the pipeline:
- Chris discovers a new manufacturer IFU or training document
- A new PubMed paper is published on an existing product
- A new YouTube channel with manufacturer content is identified
- A new industry article or society guideline is released
- Podcast corpus mining surfaces an idea that needs published backup

Input: Raw source material (PDF, URL, video, document)
Output: Source registered with type classification

**Section 3: Step 2 — Classify**

Apply SOURCE_CLASSIFICATION.md decision tree:
1. Identify source type (FDA, PubMed, manufacturer, industry, YouTube, podcast, etc.)
2. Determine layer (production vs research-only)
3. Determine what the source is authoritative for
4. Record citation format per Citation Format Reference table

If podcast/research-only: follow PODCAST_WORKFLOW.md (extract idea → find published backup)
If production-citable: proceed to Step 3

Output: Source classified with type, layer, authority scope, citation format

**Section 4: Step 3 — Chunk into Citable Units**

Break the source into chunks that can be individually cited and searched:
- PubMed: abstract + key findings (typically 1-3 chunks per paper)
- FDA label: section-based chunks (indications, dosing, warnings, contraindications — each is a chunk)
- IFU: section-based chunks (technique, storage, reconstitution, warnings)
- YouTube: transcript chunks with timestamps (start_seconds, end_seconds per chunk)
- Industry articles: paragraph-level chunks with section context
- Manufacturer training: topic-based chunks (product handling, injection technique, patient selection)

Each chunk must carry:
- Source metadata (type, title, author/manufacturer, year, URL)
- Citation data (PMID, DOI, page number, timestamp — whatever applies per citation format)
- Offering association (which gl_products product does this relate to?)

Output: Array of citable chunks with metadata

**Section 5: Step 4 — Make Searchable (Vector DB)**

Embed chunks and store in the appropriate vector collection:
- CMS Supabase project `gjqicqldjgvrwmtkliie` hosts vector collections
- PubMed chunks: `match_pubmed_articles()` RPC
- Podcast chunks: `match_podcast_chunks()` RPC
- Manufacturer/IFU chunks: new collection needed (see MANUFACTURER_DATA_ACCESS.md)
- Industry articles: existing 87K chunk corpus

Each embedded chunk must be retrievable by:
- Semantic search (vector similarity)
- Product filter (offering_id or product name)
- Source type filter (to retrieve only FDA sources, only PubMed, etc.)

Output: Chunks embedded and searchable via RPC functions

**Section 6: Step 5 — Mark Affected Fuel Docs for Review**

When new source data is added for a product:
1. Identify which fuel documents reference that product (query `gl_agent_fuel_documents` by offering_id)
2. Mark those fuel docs as `needs_review: true` (or equivalent stale flag — design in Phase 14)
3. Record what changed: "New PubMed paper added for Sculptra — fuel doc may need updated clinical evidence"

This does NOT automatically regenerate fuel docs. It flags them for the next regeneration cycle.

Output: Affected fuel docs flagged for review

**Section 7: Concrete Walkthrough — Adding a New Galderma IFU**

Walk through the full pipeline with a real example:

1. **Add:** Chris obtains the Restylane Lyft Instructions for Use (PDF) from Galderma
2. **Classify:** Source type = Manufacturer Clinical Data. Layer = Production. Authoritative for = product-specific technique, reconstitution, storage, injection depths, patient selection. Citation format = `[Galderma, Restylane Lyft IFU, 2024]`
3. **Chunk:** Break IFU into sections: Indications (1 chunk), Injection Technique (2-3 chunks), Reconstitution/Storage (1 chunk), Warnings/Precautions (1 chunk), Clinical Studies (1-2 chunks). Each chunk carries: source_type=manufacturer_ifu, manufacturer=Galderma, product=Restylane Lyft, year=2024, page_ref=p.{N}
4. **Vector DB:** Embed chunks into manufacturer data collection. Associate with Restylane Lyft offering_id. Searchable via `match_manufacturer_docs()` RPC (to be created).
5. **Mark stale:** Query `gl_agent_fuel_documents WHERE offering_id = {restylane_lyft_id}`. Flag any existing fuel docs for review.

**Section 8: Concrete Walkthrough — Adding a New YouTube Channel**

1. **Add:** Identify Allergan Aesthetics official YouTube channel
2. **Classify:** Source type = YouTube (manufacturer channel). Layer = Production. Authoritative for = product demonstrations, technique education. Citation = `[Allergan Aesthetics, "{title}", {year}]` + URL with timestamp
3. **Chunk:** Transcribe videos. Break into topic-based chunks with start_seconds/end_seconds timestamps. Each chunk carries: source_type=youtube_manufacturer, channel=Allergan Aesthetics, video_title, year, url with timestamp
4. **Vector DB:** Embed into manufacturer data collection (or separate YouTube collection). Associate with relevant offering_ids.
5. **Mark stale:** Flag affected fuel docs.

**Section 9: What This Pipeline Does NOT Do**

- Does NOT auto-generate fuel docs (that is Phase 12-14 work, requiring human review)
- Does NOT execute SQL (sources are vector-searchable, not SQL inserts)
- Does NOT replace the PODCAST_WORKFLOW.md for podcast sources (that workflow is a prerequisite sub-loop within Step 2)
- Does NOT build the Evidence Ask UI (separate workstream, out of scope)
- Does NOT implement practice-level source overrides (post-v1.2)

**Section 10: Related Documents**
- SOURCE_CLASSIFICATION.md — Source type classification and citation formats
- EVIDENCE_MODEL.md — Two-layer model, contamination definition
- PODCAST_WORKFLOW.md — Sub-workflow for podcast-type sources
- MANUFACTURER_DATA_ACCESS.md — How manufacturer data is made searchable for agents

# Manufacturer Data Access: Agent Runtime Search Design

**Phase:** 11 — Source Framework & v1.1 Close-Out
**Status:** Design document — implementation in future phases
**Requirement:** SRCE-04
```

**Section 1: Problem Statement**

Agents need to answer product-specific questions that go beyond what fuel docs cover:
- "What is the reconstitution ratio for Sculptra?"
- "What injection depth does the Restylane Lyft IFU specify for cheek augmentation?"
- "What are the storage requirements for Daxxify?"
- "What clinical trial data supports Vollure XC for nasolabial folds?"

Fuel docs (Phase 12-14) cover common cases. Agent tools search manufacturer data at runtime for the long tail — specific IFU references, precise dosing tables, storage requirements, technique details.

**Section 2: Data Sources**

Manufacturer data encompasses:
1. **Instructions for Use (IFUs)** — Official product usage documents. Source of truth for technique, dosing, reconstitution, storage, warnings.
2. **Package Inserts (PIs)** — Regulatory documents (overlap with FDA labels). Source of truth for indications, contraindications, adverse events.
3. **Manufacturer Training Materials** — Videos, slide decks, technique guides. Source for injection technique, product handling, patient selection.
4. **Clinical Study Reports** — Manufacturer-sponsored trials. Source for efficacy data, patient outcomes, study populations.
5. **Product Education Content** — Marketing-adjacent but factual content about mechanisms, benefits, differentiators.

Per SOURCE_CLASSIFICATION.md, all of these are production-citable for product education and technique guidance.

**Section 3: Access Pattern**

The target access pattern for agents:

```
Agent receives question about product X
  → Agent calls tool: search_manufacturer_data(product_id, query)
  → Tool searches manufacturer data vector collection
  → Returns top-K chunks with source metadata + citation info
  → Agent synthesizes answer with proper citation format
```

This is the same pattern as existing `match_pubmed_articles()` and `match_podcast_chunks()` RPCs in CMS Supabase.

**Section 4: Required Infrastructure (Not Built Yet)**

What needs to exist for SRCE-04:

1. **Vector collection for manufacturer data** in CMS Supabase (`gjqicqldjgvrwmtkliie`):
   - Table: `manufacturer_chunks` (or similar)
   - Columns: id, offering_id, source_type (ifu/pi/training/clinical_study), manufacturer, document_title, year, chunk_text, chunk_embedding (vector), page_ref, url, created_at
   - RPC: `match_manufacturer_docs(query_embedding, match_threshold, match_count, offering_id_filter)`

2. **Populated chunks** for the 18 demo products:
   - Chris is populating manufacturer documents in parallel (per ROADMAP.md Phase 12 note)
   - Pipeline from ENRICHMENT_PIPELINE.md Step 3-4 processes these into chunks

3. **Agent tool wrapper** in the A360 platform:
   - Tool name: `search_manufacturer_data`
   - Input: product name or offering_id, natural language query
   - Output: ranked chunks with citation metadata
   - Registered in agent tool registry

**Section 5: Current State**

What exists today:
- CMS Supabase has `match_podcast_chunks()` and `match_pubmed_articles()` — manufacturer data follows the same pattern
- 87K industry article chunks are already embedded and searchable
- Chris is gathering manufacturer documents (IFUs, training materials) in parallel

What does NOT exist:
- No manufacturer_chunks table or collection
- No `match_manufacturer_docs()` RPC
- No agent tool wrapper for manufacturer search
- No chunked manufacturer data (documents exist but are not chunked/embedded)

**Section 6: Implementation Phases**

This document defines the DESIGN. Implementation happens in later phases:

| Step | What | When |
|------|------|------|
| Chris gathers IFUs/training docs | Manual collection | In parallel now |
| Create manufacturer_chunks table + RPC | Schema + function creation | Phase 12 prep or standalone task |
| Chunk + embed documents per ENRICHMENT_PIPELINE.md | Processing pipeline | After documents collected |
| Register agent tool | Platform integration | After chunks searchable |

**Section 7: Interim Access (Before Full Implementation)**

Until the manufacturer data vector collection exists, agents can:
1. Reference fuel docs (which synthesize manufacturer data as part of their content)
2. Reference evidence_links with source='manufacturer' (existing rows from Phase 2 dossier compilation)
3. Reference FDA labels via existing evidence_links (overlap with manufacturer PIs)

This is sufficient for Phase 12-13 fuel doc generation. Full runtime search (SRCE-04 complete) enhances agent capability but is not a blocker for fuel doc creation.
