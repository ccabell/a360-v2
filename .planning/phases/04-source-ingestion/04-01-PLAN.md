1. **Count and list source_registry rows by status:**


2. **List all review rows (the triage workload):**
```sql
SELECT id, name, publisher, source_kind, authority_rank, rights_class, ingestible, status, added_by
FROM source_registry WHERE status = 'review' ORDER BY authority_rank ASC, name ASC;
```

3. **Detect duplicate names:**
```sql
SELECT name, COUNT(*) as n FROM source_registry GROUP BY name HAVING COUNT(*) > 1 ORDER BY n DESC;
```

4. **Verify ingestion_queue column schema** (pitfall 3 from research — column names may differ from spec):
```sql
SELECT column_name, data_type, is_nullable FROM information_schema.columns
WHERE table_name = 'ingestion_queue' ORDER BY ordinal_position;
```

5. **List all queued ingestion_queue items:**
```sql
SELECT iq.*, sr.name as source_name, sr.rights_class, sr.ingestible
FROM ingestion_queue iq
LEFT JOIN source_registry sr ON sr.id = iq.source_id
WHERE iq.status = 'queued'
ORDER BY sr.authority_rank ASC;
```
(If the join column is `source_registry_id` instead of `source_id`, adjust accordingly — pitfall 3.)

6. **List evidence_links with source='fda_label' that have NULL or empty url:**
```sql
SELECT el.id, el.offering_id, el.source, el.url, o.name as product_name
FROM evidence_links el
LEFT JOIN offerings o ON o.id = el.offering_id
WHERE el.source = 'fda_label' AND (el.url IS NULL OR el.url = '')
ORDER BY o.name;
```

Using the audit results from Task 1, create `supabase/compile_sql/04-01-source-registry-triage.sql` with the complete triage. The SQL file must be idempotent (safe to re-run).

**Section 1: Deduplicate source_registry**
For any duplicate names found in Task 1, keep the row with the most complete data (non-null publisher, base_url, doi_prefix). Delete the other(s). Use:
```sql
-- For each duplicate name, keep the one with the most populated fields
-- DELETE FROM source_registry WHERE id = '<duplicate-uuid>' AND name = '<name>';
```

**Section 2: Promote ingestible sources to 'active'**
These are the sources whose rights_class makes them ingestible (public_domain, open_access_cc_by, manufacturer_permitted). The `ingestible` generated column will already be TRUE for these. Promote them:
```sql
UPDATE source_registry SET status = 'active'
WHERE status = 'review' AND rights_class IN ('public_domain', 'open_access_cc_by', 'manufacturer_permitted');
```

**Section 3: Reject non-ingestible sources with reason**
For paywalled, open_access_cc_by_nc, and society_guideline sources, reject with a note column if one exists, or just set status. Check if source_registry has a `notes` or `rejection_reason` column first.

Classification rules (from research):
- `paywalled` -> reject: "Subscription journal — cite-only per rights gate"
- `open_access_cc_by_nc` -> reject: "CC-BY-NC — commercial restriction applies to A360; cite-only"
- `society_guideline` -> reject: "Society guideline — copyrighted; cite-only"
- `unknown` -> Case-by-case. If the journal is clearly paywalled (J Cosm Derm, Ann Plast Surg, etc.), update rights_class to 'paywalled' first then reject. If it's a hybrid journal, set rights_class='unknown' and status='active' with a note that per-article verification is needed.

```sql
-- Reject paywalled
UPDATE source_registry SET status = 'rejected'
WHERE status = 'review' AND rights_class = 'paywalled';

-- Reject CC-BY-NC
UPDATE source_registry SET status = 'rejected'
WHERE status = 'review' AND rights_class = 'open_access_cc_by_nc';

-- Reject society guidelines
UPDATE source_registry SET status = 'rejected'
WHERE status = 'review' AND rights_class = 'society_guideline';
```

For `unknown` rights_class rows: reclassify based on the AESTHETIC_DERMATOLOGY_JOURNAL_REGISTRY.md and research knowledge, then promote or reject accordingly.

**Section 4: Triage ingestion_queue**
- Rows whose source is ingestible AND the queue item is an FDA/DailyMed document -> status='approved'
- Rows whose source is paywalled/CC-BY-NC -> status='rejected' (use the actual `notes` column name from schema inspection)
- Use the actual column names confirmed in Task 1
