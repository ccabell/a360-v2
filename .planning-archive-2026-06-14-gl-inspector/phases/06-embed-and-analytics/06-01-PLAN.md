1. Create `lib/ask/analytics.ts` — a thin client-side helper:
   - Export `trackEvent(eventName: string, payload?: Record<string, unknown>): void`
   - Fire-and-forget: `fetch("/api/ask/event", { method: "POST", body: JSON.stringify({ event_name: eventName, ...payload }), keepalive: true }).catch(() => {})` — no await, no error surfaced to user
   - Export type `AnalyticsEventName = "evidence_unauth_ask" | "evidence_answer_complete" | "citation_click"`
   - The function should accept any string but the type alias documents expected values

2. Create `app/api/ask/event/route.ts` — POST-only route:
   - Accept JSON body: `{ event_name: string; surface?: string; latency_ms?: number; citation_count?: number; citation_url?: string; citation_tier?: string; question?: string }`
   - Validate event_name is one of: "evidence_unauth_ask", "evidence_answer_complete", "citation_click" — return 400 otherwise
   - Call `getOrCreateSession(req, body.surface ?? "public")` to get sessionId
   - Insert into ask_log: `{ session_id: sessionId, surface: body.surface ?? "public", question: body.question ?? event_name, status: "event", event_name: body.event_name, latency_ms: body.latency_ms ?? 0, client_ip_hash: "event" }`
   - Note: ask_log may not have an `event_name` column yet. Use the `question` field to encode it as `event:{event_name}` (matching the pattern from feedback/route.ts which uses `feedback:{signal}:cite_{N}`) AND store the event_name in a JSONB metadata field if the table supports it. Simplest: just use `question: "event:" + body.event_name` and store extra data in a `metadata` JSONB column if available, otherwise encode key data in the question string like: `event:citation_click:url={url}:tier={tier}`
   - Return `{ ok: true }` with 200

3. Add `event_name` tagging to server-side ask_log inserts in `app/api/ask/route.ts`:
   - On the existing insert at ~line 380 (status: "complete", live path): add `question` field already has `q` — append nothing, but add a comment noting this row = evidence_unauth_ask + evidence_answer_complete (the server log IS the ask event + answer event for server-side tracking)
   - Actually, the simplest approach: the server-side route already logs every ask with latency_ms and citations. For ANLY-01 and ANLY-02, the client-side trackEvent calls will create ADDITIONAL rows with explicit event names. The server row with `status: "complete"` IS the canonical answer record; the event rows are supplementary analytics markers.
