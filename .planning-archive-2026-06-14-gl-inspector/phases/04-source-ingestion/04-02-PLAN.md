Output: FDA PDFs in Supabase Storage; all evidence_links fda_label rows have working URLs.
</objective>

```sql
SELECT el.id, el.offering_id, el.source, el.url, el.page_number, o.name as product_name
```

**Step 3: Create scripts/download-fda-pdfs.ts**

This script must:
1. Define a `FDA_PDF_MAP` mapping product names to their FDA Access Data PDF URLs
2. For each entry, attempt to fetch the PDF using the FDA API pattern from Phase 1 (NOT direct accessdata.fda.gov — that blocks robots per 01-02-SUMMARY.md)
3. Use `api.fda.gov/drug/drugsfda.json` for NDA/BLA products to find the current label PDF URL
4. Use `api.fda.gov/device/pma.json` or `api.fda.gov/device/510k.json` for device products
5. Upload each PDF to Supabase Storage bucket under path `fda/{application_number}.pdf`
6. Print a report of what was uploaded and any failures

Important: FDA Access Data blocks direct curl. The script should construct URLs from API metadata, not try to download directly from accessdata.fda.gov. If direct download fails, log the URL for manual download and still create the evidence_links row pointing to the accessdata.fda.gov URL (the current working pattern from Phase 1).

Use the same patterns as `scripts/backfill-fda-urls.ts`: dotenv for env loading, Supabase client from `@supabase/supabase-js`.

**Fallback:** If Supabase Storage upload is not feasible (bucket not configured, permissions issue), skip the storage step and proceed to Task 2 with just URL verification/backfill. The primary goal is working FDA URLs in evidence_links, not necessarily local PDF copies.

**For Phase 2 products missing evidence_links fda_label rows:**
Insert new evidence_links rows. Use the idempotent WHERE NOT EXISTS pattern:

```sql
INSERT INTO evidence_links (offering_id, source, url, authority_rank, page_number)
SELECT
  '<offering_uuid>'::uuid,
  'fda_label',
  '<fda_url>',
  1,  -- FDA is always authority_rank 1
  NULL
WHERE NOT EXISTS (
  SELECT 1 FROM evidence_links
  WHERE offering_id = '<offering_uuid>' AND source = 'fda_label'
);
```

The offering_id UUIDs must come from the live DB query in Task 1 Step 1. Do NOT hardcode UUIDs without confirming them first.
