# Phase 11.1 Plan 01: Fuel Doc Type System, Data Access, and Serializer Summary

**One-liner:** TypeScript discriminated union on fuel_type (combination/product/concern) with agentSupabase CRUD, COALESCE practice resolution, and markdown serializer for agent consumption.

## What Was Built

### Task 1 — Fuel Doc TypeScript Types (`lib/types/fuel-docs.ts`)

Commit: `60f44b1`

Defined the canonical type system for the fuel doc template system:

- `FuelDocType` = `"combination" | "product" | "concern"` — the discriminant
- `ReviewStatus` = `"draft" | "in_review" | "approved" | "active"`
- `FuelDocBase` — shared fields across all 3 types (id, fuel_type, template_version, product_name, source_type, reference_id, review_status, audience, patient_safe, last_reviewed, reviewed_by, content, metadata, created_at, updated_at)
- `CombinationContent` — 28 fields from Template 1 in FUEL_DOC_TEMPLATE_SPEC.md (identity, why together, what to say, what not to say, objections, timing, maintenance, evidence)
- `ProductContent` — 20 fields from Template 2 (identity, clinical profile, talking points, FAQ, timing, evidence)
- `ConcernContent` — 16 fields from Template 3 (patient language, treatment arc, consultation language, timing, evidence)
- `SopFields` and `PreferenceFields` — practice-level-only operational fields (no GL fallback)
- `CombinationFuelDoc`, `ProductFuelDoc`, `ConcernFuelDoc` — discriminated union members
- `FuelDoc` = `CombinationFuelDoc | ProductFuelDoc | ConcernFuelDoc` — the union type
- `PracticeFuelDocOverride` — pl_agent_fuel_documents row shape
- `ResolvedFuelDoc` — post-COALESCE doc with sop, preferences, has_practice_override
- `createEmptyContent(type)` — returns typed empty defaults per fuel_type

### Task 2 — Data Access Layer (`lib/api/fuel-docs.ts`)

Commit: `ed4fcf9`

6 functions following the exact pattern from `lib/api/agents.ts`:

- `listFuelDocs(filters?)` — queries `gl_agent_fuel_documents`, filters by source_type (fuel_type), review_status, product_name ilike
- `getFuelDoc(id)` — single doc by id
- `createFuelDoc(doc)` — insert with select().single()
- `updateFuelDoc(id, updates)` — update with updated_at timestamp, select().single()
- `deleteFuelDoc(id)` — delete, returns void
- `getResolvedFuelDoc(id, practiceId?)` — implements COALESCE: fetches GL doc, optionally fetches pl_agent_fuel_documents row, merges content (pl wins), returns ResolvedFuelDoc with sop/preferences/has_practice_override

All functions use `agentSupabase` (project aejskvmpembryunnbgrk). COALESCE implemented in TypeScript via object spread: `{ ...gl.content, ...pl.content }`.

### Task 3 — Markdown Serializer (`lib/fuel-docs/serialize.ts`)

Commit: `297bfca`

Pure function `fuelDocToMarkdown(doc: ResolvedFuelDoc): string`:

- Dispatches on `doc.fuel_type` to type-specific renderers
- **Combination renderer:** # title, > positioning, ## Identity, ## Why Together, ### Clinical Rationale, ## What to Say, ## What Not to Say, ## Objection Handling, ## Timing & Logistics, ## Maintenance, ## Evidence
- **Product renderer:** # title, ## Identity, ## Clinical Profile, ## What to Say, ## What Not to Say, ## Timing, ## Evidence
- **Concern renderer:** # title, ## What Patients Say, ## Treatment Arc, ## What to Say, ## What Not to Say, ## Evidence
- Appends `## Practice SOPs` section when `doc.sop` is non-null
- Appends `## Practice Preferences` section when `doc.preferences` is non-null
- Empty fields and empty arrays skipped — headings omitted for empty sections
- `renderBulletList()` and `renderFaqList()` helpers
- Exhaustive switch with never type check

## Deviations from Plan

None — plan executed exactly as written.

The plan's verification snippet used `createEmptyContent("combination")` which produces all-empty fields, so the empty-content result shows only `# Test Combo\n` (all section headings skipped). This is the correct behavior per spec ("Skip empty string fields and empty arrays — do not render headings for sections with no content"). The behavioral check was adjusted to use populated content which correctly produces all expected headings.

None. These are type/API/serializer files — no UI rendering and no data stubs. Content fields in `createEmptyContent()` default to empty strings/arrays, which is correct (these are structure defaults, not stub placeholders — content is populated by Phase 12/13 enrichment pipelines).
