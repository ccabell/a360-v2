# Phase 11.1: fuel-doc-templates-and-management-ui - Research

**Researched:** 2026-06-14
**Domain:** Fuel doc schema design + Next.js 16 internal admin UI
**Confidence:** HIGH (project codebase verified directly; schema confirmed from migrations and type files)

## Summary

Phase 11.1 bridges two concerns that must be designed together: (1) the **template schema** that defines what a fuel doc contains, and (2) the **management UI** that lets internal users create, view, edit, and publish those fuel docs. A detailed template spec already exists in `.planning/phases/999.1-agent-fuel-document-management-ui/FUEL_DOC_TEMPLATE_SPEC.md` — this is the primary artifact the planner should use as a starting point. That spec defines three document types (combination, concern, product), field groupings, COALESCE resolution logic, and a conceptual UI wireframe.

The existing `gl_agent_fuel_documents` table stores fuel docs as flat JSONB rows with two incompatible formats (Format A: flat key-value, Format B: archetype-nested). Phase 11.1 must define a single canonical template and add a `pl_agent_fuel_documents` practice-override table that follows the same gl/pl COALESCE pattern used by `gl_products`/`pl_products`. The unified template will be the source of truth that Phase 12 (combination content) and Phase 13 (concern content) populate into.

The management UI fits neatly into the existing Next.js 16 App Router dashboard at `/dashboard/`. The codebase already has a working pattern for exactly this kind of internal admin tool: the Agent Manager (`/dashboard/agents/`) with list page, detail page with tabs, dialogs for creation, and a `DataTable` component. The fuel doc UI should follow this exact pattern — it is internal-use only, does not need auth changes, and the sidebar already has a `scope: "internal"` mechanism to hide it from demo mode.

**Primary recommendation:** Reuse the Agent Manager UI pattern (list → detail tabs → CRUD dialogs) for the fuel doc UI; implement the unified template as a defined TypeScript type backed by a Supabase JSONB `content` column with a versioned `template_version` field; and define the practice override table in a migration that follows the existing COALESCE pattern.

- Templates must produce MD files consumable by agents.
- Phase 12 and Phase 13 DEPEND on this phase's templates to generate content — the template structure is load-bearing for downstream phases.

### TypeScript Type for Unified Template

The canonical TypeScript type must be defined in `lib/types/fuel-docs.ts`. It formalizes the YAML spec from `FUEL_DOC_TEMPLATE_SPEC.md` into TypeScript. Key structural decision: use a **discriminated union** on `fuel_type`:

```typescript
// lib/types/fuel-docs.ts
export type FuelDocType = "combination" | "product" | "concern"
export type ReviewStatus = "draft" | "in_review" | "approved" | "active"

export interface FuelDocBase {
  id: string
  fuel_type: FuelDocType
  template_version: string
  reference_id: string | null
  review_status: ReviewStatus
  audience: string[]
  patient_safe: boolean
  last_reviewed: string | null
  reviewed_by: string | null
  created_at: string
  updated_at: string
}

export interface CombinationFuelDoc extends FuelDocBase {
  fuel_type: "combination"
  content: CombinationContent
}

// CombinationContent, ProductContent, ConcernContent match the YAML spec
```

### Anti-Patterns to Avoid

- **Duplicating field storage**: Do NOT put the same field in both a column AND inside a JSONB blob. Choose one representation.
- **Changing existing gl_agent_fuel_documents rows** during schema migration: Existing 29 rows use two JSON formats. The migration should add new columns WITHOUT touching existing `content`. Phase 12 reconciles the content (COMBO-05).
- **Requiring auth changes for the UI**: This is internal. The app already operates without per-user auth at the dashboard level.
- **New sidebar state management**: The sidebar already has `scope: "internal"` pattern — use it, don't reinvent.

## Common Pitfalls

### Pitfall 1: Breaking Existing Fuel Doc Rows

**What goes wrong:** A schema migration that alters `content` column type or adds NOT NULL constraints breaks the 29 existing rows in `gl_agent_fuel_documents`.

**Why it happens:** Existing rows use two incompatible JSON formats (Format A and Format B from the spec). Neither matches the new unified template.

**How to avoid:** All new schema columns must have DEFAULT values or be nullable. Do NOT add NOT NULL without a backfill. Do NOT modify the existing `content` column. Phase 12 handles content reconciliation.

**Warning signs:** Migration fails with "violates not-null constraint" or "invalid input for type" on existing rows.

### Pitfall 2: Conflating Template Definition with Content Population

**What goes wrong:** The plan tries to populate fuel doc content (corpus mining, AI generation) in this phase instead of just defining the schema.

**Why it happens:** The spec includes rich example content that looks like it should be filled in.

**How to avoid:** Phase 11.1 defines structure only. SOPs/preferences start blank. Clinical intelligence content comes from Phase 12 (combination) and Phase 13 (concern).

**Warning signs:** Tasks that involve "mining the corpus" or "generating with AI" — those belong to Phase 12/13.

### Pitfall 3: GL Supabase vs Agent Supabase Confusion

**What goes wrong:** Writing fuel doc API routes that use the wrong Supabase client.

**Why it happens:** The app has THREE Supabase clients (`agentSupabase`, `cmsSupabase`, `prSupabase` in `lib/supabase.ts`). The GL data (products, fuel docs) lives in the Agent Supabase project, NOT a "GL Supabase" — it's all in `agentSupabase` (project `aejskvmpembryunnbgrk`).

**How to avoid:** Use `agentSupabase` for all fuel doc queries. This is the same client used by `lib/api/agents.ts`.

**Warning signs:** Fuel doc queries returning empty results despite data existing in the DB.

### Pitfall 4: MD Serialization Overengineering

**What goes wrong:** Building a complex templating system with a template engine or DSL.

**Why it happens:** "Template system" sounds like it needs a template engine.

**How to avoid:** The serialization is a plain TypeScript function. The "template" is the TypeScript type, not a file-based template. The output is a markdown string built by string interpolation.

### Pitfall 5: Next.js 16 API Route Pattern

**What goes wrong:** Writing API routes using Next.js 13/14 patterns that have changed in Next.js 16.

**Why it happens:** Training data is based on Next.js 13/14. The AGENTS.md file explicitly warns about breaking changes.

**How to avoid:** Copy the existing API route pattern from `app/api/agents/route.ts` exactly. Do not invent new patterns. The existing routes work correctly in Next.js 16.

import { agentSupabase } from "@/lib/supabase";

export async function listFuelDocs(filters?: { type?: FuelDocType; status?: ReviewStatus }) {
  let query = agentSupabase
    .from("gl_agent_fuel_documents")
    .select("*")
    .order("updated_at", { ascending: false });

  if (filters?.type) query = query.eq("fuel_type", filters.type);
  if (filters?.status) query = query.eq("review_status", filters.status);

  const { data, error } = await query;
  if (error) throw error;
  return data;
}

## Key Architecture Decision: Template as TypeScript Type, Not File Template

The phrase "template system produces MD files" in the success criteria needs clarification. The planner should understand this as:

1. **Template = TypeScript discriminated union type** (`CombinationFuelDoc | ProductFuelDoc | ConcernFuelDoc`) stored in `lib/types/fuel-docs.ts`
2. **Content = JSONB in Supabase** following the typed structure
3. **MD output = serialization function** `fuelDocToMarkdown(resolvedDoc)` that renders the COALESCED content as structured markdown
4. **Agents consume** the markdown string returned from `get_fuel_document` tool

This is not a file-based template system. There are no `.md.template` files. The "template" is the type definition, and the "output" is the serialized markdown string.

## Open Questions

1. **Does `gl_agent_fuel_documents` already have a `fuel_type` column, or only `source_type`?**
   - What we know: `lib/types/data-sources.ts` shows `source_type: string` (not `fuel_type`)
   - What's unclear: Whether `source_type` is being used as `fuel_type` already, or if they're different things
   - Recommendation: Check live DB schema before writing migration. Use `agentSupabase.from('gl_agent_fuel_documents').select('*').limit(1)` in a script to inspect actual column names. If `source_type` IS the fuel type discriminator, rename plan accordingly.

2. **Should the practice override use a separate JSONB `content` column or field-by-field columns?**
   - What we know: The spec uses `COALESCE(pl.content->>'field', gl.content->>'field')` — JSONB approach
   - What's unclear: Whether JSONB extraction vs column-per-field matters for query performance at this scale
   - Recommendation: Use JSONB for practice overrides (matches GL pattern, allows sparse overrides where practice only overrides some fields)

3. **Where exactly does `pl_agent_fuel_documents` live — same DB as `gl_agent_fuel_documents`?**
   - What we know: GL Supabase = agent Supabase (project `aejskvmpembryunnbgrk`), accessed via `agentSupabase` client
   - Recommendation: Yes, same DB. Practice override table lives alongside GL table. The practice_id FK needs to resolve to a practices table in that DB — verify it exists before writing the migration.
